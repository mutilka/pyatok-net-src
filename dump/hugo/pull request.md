Сообщение пользователя
Мой репозиторий, коллега соавтор как ограничить его права на пуш в мэйн
Ответ Grok 3
Чтобы ограничить права коллеги-соавтора (collaborator) на пуш в ветку main в вашем репозитории на GitHub, нужно настроить защиту ветки (branch protection). Это стандартный способ в GitHub, чтобы контролировать, кто и как может вносить изменения в ключевые ветки. Вот пошаговая инструкция:
Шаги по ограничению прав на пуш в main
Перейдите в настройки репозитория:
Откройте ваш репозиторий на GitHub.

Нажмите на вкладку Settings (Настройки) в верхней части страницы.
Найдите раздел управления ветками:
В левом меню выберите Branches (Ветки).
Настройте защиту ветки:
В разделе Branch protection rules (Правила защиты веток) нажмите Add rule (Добавить правило).

В поле Branch name pattern укажите имя ветки, например, main (или master, если у вас старая основная ветка).
Включите защиту:
Поставьте галочку напротив Protect matching branches (Защитить соответствующие ветки).
Ограничьте прямой пуш:
Активируйте опцию Restrict who can push to matching branches (Ограничить, кто может пушить в соответствующие ветки).

В появившемся списке выберите только себя (владельца репозитория) или конкретных пользователей/команды, которым вы хотите разрешить пушить напрямую. Ваш коллега-соавтор в этом списке указан быть не должен.
Дополнительные настройки (опционально):
Require a pull request before merging (Требовать pull request перед слиянием) — включите эту опцию, чтобы изменения в main вносились только через pull request. Это заставит коллегу работать через PR, которые вы сможете проверять.

Require approvals (Требовать одобрения) — укажите, что для слияния PR нужно ваше одобрение.

Restrict who can dismiss pull request reviews (Ограничить, кто может отклонять ревью PR) — выберите только себя, если хотите полный контроль.
Сохраните изменения:
Нажмите Create или Save changes внизу формы.
Что это даст?
Ваш коллега-соавтор не сможет напрямую пушить в ветку main.

Ему придется создавать свои ветки (например, feature/его-изменения), пушить изменения туда, а затем открывать pull request в main.

Вы сможете просмотреть и одобрить (или отклонить) его изменения перед слиянием.
Важные замечания
Если репозиторий принадлежит организации, а не вашему личному аккаунту, у вас будет больше гибкости в настройке прав через команды (teams). Для личного репозитория таких опций меньше, но защита ветки работает одинаково.

Даже если коллега — соавтор (collaborator), он не сможет обойти защиту ветки, если его явно не указали в разрешенных для пуша.
Таким образом, вы сохраните контроль над веткой main, а коллега сможет вносить изменения только через процесс ревью. Если что-то непонятно или нужно уточнить, дай знать!
